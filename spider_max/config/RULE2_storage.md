# 规则二：三层存储与知识检索规范

> 适用对象：数据科学家、企业医生、技术专家、文档智能处理模块  
> 核心目标：严格执行热/温/冷数据分离，保障检索性能（P99 < 500ms）。

---

## 1. 数据分层路由策略

### 1.1 智能路由注解
在检索代码中，必须使用 `@route_vector_store` 装饰器或策略模式，
根据 `data_type` 参数自动选择数据库：

| 层级 | 存储介质 | 适用数据 | 响应时间 |
|------|----------|----------|----------|
| **Hot** (热数据) | Chroma (D盘, 内存优先) | 实时对话、高频访问 | 微秒级 |
| **Warm** (温数据) | Qdrant (E盘, SSD存储) | 近期项目、周级数据 | 毫秒级 |
| **Cold** (冷数据) | Milvus (F盘, 高吞吐) | 历史归档、月度报表 | 秒级 |

```python
@route_vector_store(data_type="hot")  # 自动路由到 Chroma
def search_hot_knowledge(query: str):
    ...
```

### 1.2 自动升降级
- 代码中**必须**包含数据热度检测逻辑
- 当冷数据访问频率 > 10次/小时，自动迁移至温层
- 当数据时间进入最近30天，自动迁移至热层
- 降级规则：30天未访问的热数据 → 温层 → 冷层

## 2. 检索性能与安全红线

### 2.1 Token 限制
- 所有检索上下文拼接后的 Token 数**严禁**超过模型限制的 80%
- 例如：32K 上下文模型，检索内容 ≤ 25,600 tokens
- 超出时自动截断，并记录 Warning 日志

### 2.2 混合检索
- **严禁**仅使用向量检索
- 必须结合以下至少一种策略提升准确率：
  - **MMR**（最大边际相关性）：平衡相关性与多样性
  - **HyDE**（假设性文档嵌入）：先生成假设文档再检索
  - **混合搜索**：关键词 + 向量检索（BM25 + Vector）

### 2.3 MCP 集成安全
- GitHub、Obsidian 等 MCP 服务的 Token 必须通过环境变量注入
- **严禁**在代码中硬编码 Token 或密钥
- 默认连接为**只读模式**
- 如需写入操作（如自动提交代码），必须经过 `SystemManager` 的二次确认
